# 프론트엔드 인증/토큰 로직 현황

백엔드 플로우(토스 로그인 → 이메일 전송 → 이메일 인증 → 2차 토큰 → 프로필 설정)와 비교한 **현재 프론트 상태** 요약입니다.

---

## 백엔드 플로우 (참고)

| 단계 | 백엔드 기대 | 프론트 현황 |
|------|-------------|-------------|
| 1 | 토스 로그인 → tossUserKey로 임시 유저 생성 → **1차 임시 토큰** 발급 | ❌ 미구현 |
| 2 | 이메일 전송 API (1차 토큰으로 식별) | ❌ 미구현 |
| 3 | 이메일 인증 API (인증 코드 + **1차 토큰** 전송) | ❌ 미구현 |
| 4 | 인증 완료 → **2차 임시 토큰** 발급 (tossUserKey, email, verified) | ❌ 미구현 |
| 5 | 프로필 설정 API (**2차 토큰** 전송) | ❌ 미구현 |

---

## 1. 토스 로그인 & 1차 토큰 (미구현)

- **위치:** `app/page.tsx` → `SplashScreen`
- **현재:** 스플래시 2.8초 후 `/verify`로 이동만 함. **토스 로그인/브릿지 호출 없음**, tossUserKey·1차 임시 토큰 수신·저장 없음.
- **필요:**  
  - 앱인토스 브릿지로 토스 로그인 후 tossUserKey 수신  
  - 백엔드 로그인 API 호출 → **1차 임시 토큰** 수신  
  - 해당 토큰 저장 (예: `localStorage`/메모리) 후 이후 단계에서 사용

---

## 2. 토큰 저장소 (미구현)

- **현재:** `localStorage`, `sessionStorage`, 쿠키, 전역 상태 등 **어디에도 토큰을 저장/조회하는 코드 없음**.
- **필요:**  
  - 1차 임시 토큰 저장 (이메일 전송·인증 시 사용)  
  - 2차 임시 토큰 저장 (프로필 설정·이후 인증 필요 API에 사용)  
  - (선택) 토큰 만료/갱신 처리

---

## 3. 이메일 전송 (목업만 존재)

- **위치:** `components/verify-page.tsx` → `handleSendCode`
- **현재:**  
  - `setTimeout(..., 1500)` 으로 1.5초 후 `step === "code"`로만 변경  
  - **실제 API 호출 없음**, 1차 토큰 전송 없음
- **필요:**  
  - 1차 임시 토큰을 헤더(예: `Authorization: Bearer <1차 토큰>`)에 담아 이메일 전송 API 호출  
  - 성공 시 `step === "code"` 로 전환

---

## 4. 이메일 인증 (목업만 존재)

- **위치:** `components/verify-page.tsx` → `handleVerify`
- **현재:**  
  - `setTimeout(..., 1500)` 후 `step === "done"`  
  - **인증 API 호출 없음**, 1차 토큰·인증 코드 전송 없음
- **필요:**  
  - 인증 API 호출 시 **요청에 1차 임시 토큰 + 인증 코드** 포함  
  - 응답에서 **2차 임시 토큰** 수신 후 저장  
  - 그 다음 `step === "done"` 후 "프로필 설정하기"로 진행

---

## 5. 프로필 설정 (목업만 존재)

- **위치:** `components/profile-setup.tsx`
- **현재:**  
  - 폼 상태만 로컬로 관리, "시작하기!" 시 `onComplete()` → `router.push("/home")`  
  - **프로필 설정 API 호출 없음**, 2차 토큰 전송 없음
- **필요:**  
  - **2차 임시 토큰**을 헤더에 담아 프로필 설정 API 호출  
  - 성공 시 정식 토큰/세션 수신 여부에 따라 처리 후 `/home` 이동

---

## 6. API 레이어 & 인증 헤더 (미구현)

- **현재:**  
  - `fetch` / `axios` 등으로 백엔드 API를 호출하는 공통 모듈 없음  
  - `Authorization` 헤더를 붙이는 코드 없음
- **필요:**  
  - `lib/api.ts` (또는 `lib/auth.ts`) 등에서  
    - 베이스 URL  
    - 저장된 토큰을 읽어 `Authorization: Bearer <token>` 설정  
  - 이메일 전송/인증/프로필 설정 등에서 이 클라이언트 사용

---

## 7. 라우트 가드 (미구현)

- **현재:**  
  - `middleware.ts` 없음  
  - `/verify`, `/profile-setup`, `/home` 등 **토큰/로그인 여부와 무관하게 접근 가능**
- **필요 (선택):**  
  - 1차 토큰 없으면 `/verify` 이전 단계로 리다이렉트  
  - 2차 토큰(이메일 인증 완료) 없으면 `/profile-setup` 이전으로 리다이렉트  
  - 정식 로그인 없으면 `/home` 등 메인 기능 접근 제한

---

## 8. 플로우 요약

```
[현재]
/ (스플래시) → /verify (이메일·코드 UI만, setTimeout 목업) → /profile-setup (폼만) → /home
토큰 없음, API 호출 없음.

[백엔드와 맞추려면]
/ (스플래시 + 토스 로그인 + 1차 토큰 저장)
  → /verify (이메일 전송 API + 인증 API, 1차→2차 토큰 교체 저장)
  → /profile-setup (2차 토큰으로 프로필 API)
  → /home
```

---

## 9. 권장 작업 순서

1. **토큰 저장/조회 유틸**  
   - 예: `lib/auth.ts` 에서 1차/2차 토큰 저장·조회 (localStorage 또는 메모리)
2. **API 클라이언트**  
   - 예: `lib/api.ts` 에서 baseURL + `Authorization` 헤더 붙이는 `fetch`/axios 래퍼
3. **토스 로그인 연동**  
   - 스플래시 또는 전용 진입점에서 토스 로그인 → 백엔드 로그인 API → 1차 토큰 저장
4. **verify-page**  
   - `handleSendCode` → 이메일 전송 API (1차 토큰)  
   - `handleVerify` → 인증 API (1차 토큰 + 코드) → 2차 토큰 저장
5. **profile-setup**  
   - 제출 시 프로필 API (2차 토큰)
6. **(선택) 라우트 가드**  
   - middleware 또는 레이아웃에서 토큰 유무에 따라 리다이렉트

이 순서대로 적용하면 백엔드에서 설계한 “토스 로그인 → 1차 토큰 → 이메일 인증 → 2차 토큰 → 프로필 설정” 플로우와 프론트가 맞게 됩니다.
